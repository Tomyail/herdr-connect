# Herdr Connect CLI

[简体中文](zh-CN/cli.md)

The `herdr-connect` CLI helps an owner check a local Herdr installation and start the same-LAN daemon. Start with:

```sh
herdr-connect doctor
herdr-connect service install
herdr-connect service status
```

Run `herdr-connect help` for the complete command list, global options, examples, and safety boundary. `help <command>`, `<command> --help`, and `<command> -h` show command-specific usage. `herdr-connect version` and `herdr-connect --version` print the release version; source builds that did not receive release metadata report `development`.

## Diagnose an installation

`doctor` is the human-readable starting point. It checks that the SQLite database can be opened and validated, that the selected Herdr source is reachable, that at least one Agent is visible, and whether TCP port `9808` is available for the LAN daemon. It ends with a command the owner can run next:

```console
$ herdr-connect doctor
Herdr Connect doctor
[OK] Database: /home/owner/.config/herdr-connect/daemon.db (schema v2)
[OK] Herdr CLI/source: herdr-cli-v0.7 is online
[OK] Agents: 2 found
[OK] LAN daemon port: TCP 9808 is available
Next: Ready. Run '/home/owner/.local/bin/herdr-connect service install' to install and start the background service.
```

If Herdr is missing or unavailable, first make `herdr agent list` work. If no Agents are found, start an Agent and retry `herdr-connect doctor`. When another Herdr Connect daemon is detected on `9808`, doctor reports that it is already running instead of asking the owner to start a duplicate. If an unrelated process owns the port, doctor exits with status 1 and suggests how to inspect it. `doctor --json` provides the same checks in a structured form for new automation.

An installed binary is shown by its executable path. A temporary `go run` binary cannot be installed as a persistent service; source development can continue to use the foreground `go run ./cmd/herdr-connect --source herdr demo-lan` command.

## Manage the background service

On macOS and Linux, install and immediately start an owner-level service explicitly:

```sh
herdr-connect service install
herdr-connect service status
```

The installer resolves both the Herdr Connect binary and the current `herdr` executable to absolute paths and stores them in the service configuration. Do not move or delete either executable while the service is installed. Override Herdr when needed with `service install --herdr /absolute/path/to/herdr`. Herdr may be temporarily offline, but the executable must exist and be executable. The command refuses to overwrite an unmanaged service file or take over an unrelated process on TCP `9808`.

The complete lifecycle is:

```sh
herdr-connect service status --json
herdr-connect service logs
herdr-connect service logs --tail
herdr-connect service restart
herdr-connect service uninstall
```

`logs` prints the latest 100 lines; `--tail` follows new output. `uninstall` stops the service and removes only its managed configuration, preserving the CLI binary, database, and logs. Re-running `service install` atomically updates a configuration previously generated by Herdr Connect. macOS uses `~/Library/LaunchAgents/com.tomyail.herdr-connect.plist`; Linux requires systemd user services and uses `~/.config/systemd/user/herdr-connect.service`. Windows service management is not supported yet.

`diagnostics` remains the compatibility command for existing scripts. Its default output is JSON, and `diagnostics --json` is an explicit equivalent. Its established fields are `database`, `schema_version`, `source_name`, `source_online`, `agent_count`, and `through_event_seq`; an unavailable source adds the stable `source_error: "source_unavailable"` marker.

## Pair a device over the LAN

`pair` issues a one-time QR code that a Herdr Connect device scans to establish its own per-device token. It requires the LAN daemon to already be running. In one terminal start the daemon, then in a second terminal run:

```sh
herdr-connect --source herdr demo-lan   # terminal 1
herdr-connect pair                       # terminal 2
```

`pair` first probes whether the daemon is listening on `9808`; if it is not, the command exits non-zero and tells you to start `demo-lan` first. While waiting it prints a scannable terminal QR whose payload carries the installation certificate fingerprint (`fp`), candidate LAN host addresses, the fixed port `9808`, and a one-time `secret`. A device that scans the QR submits the secret to `/v1/pair` to complete pairing; the command polls until pairing completes or the secret times out, then prints the paired device name and `device_id`. The plaintext device token is returned only to the pairing device and is never printed on the host.

## Commands and options

Global options must appear before the command:

```text
--source herdr|fake   Source adapter; defaults to herdr
--db PATH             SQLite path; defaults to the owner's config directory
-h, --help            Show help without contacting Herdr or opening SQLite
--version             Show the build version without contacting Herdr or opening SQLite
```

The owner-facing commands are `doctor`, `service`, `demo-lan`, `pair`, `diagnostics`, `status`, `agents`, `capabilities`, `migrations`, and `daemon`. `daemon --once` performs one synchronization for a scripted health check. `trace` and `--source fake` are development tools.

CLI output follows a stable convention:

- JSON and requested help/version output go to stdout.
- Warnings and errors go to stderr, except `doctor` check results, which are the requested report and go to stdout.
- Exit 0 means success, exit 1 means a runtime or health-check failure, and exit 2 means invalid command-line usage.
- `service status` and other lifecycle commands use exit 3 when the service is not installed.
- Help, version, unknown commands, and argument errors are handled before Herdr is contacted or SQLite is opened. Unknown commands include a suggestion when a close command name exists.

## LAN-only security boundary

`demo-lan` listens on TCP port `9808` and advertises `_herdr-connect._tcp`. Transport is HTTPS with a self-signed ECDSA P-256 certificate whose SHA-256 fingerprint is the installation identity that devices pin. Every endpoint except `/v1/pair` requires an `Authorization: Bearer <token>` from a paired device; missing/unknown tokens receive `401 unauthorized`, and revoked devices receive `401 revoked`. Pairing is a single-use QR secret exchanged at `/v1/pair` for a per-device token. Recent terminal output and text input are only reachable through these authenticated endpoints.

The supported product scope is local-first LAN control: keep the daemon on networks and VPNs you operate, do not expose TCP `9808` directly to the public internet, and revoke devices you no longer use. Discovery proves reachability only; trust is established by QR pairing and certificate pinning. See [LAN TLS and pairing](security/lan-tls-pairing.md) for the trust model, certificate lifecycle, and pairing flow.

For binary installation and platform notes, see the [daemon installation guide](release/daemon.md).
