# Herdr Connect Protocol v1

状态：已冻结，可用于 MVP 实现。

Protocol v1 定义 daemon、Device 与 Relay 之间与传输无关的消息格式。LAN 和 Cloudflare Relay 必须承载完全相同的 envelope、标识符、密码语义、ACK 和重放规则；WebSocket、Durable Objects、mDNS、Expo Push 等运行时细节不属于本协议。

## 设计目标

- Relay 与局域网观察者不能读取 prompt、Agent 输出或生命周期内容。
- 每个 Device 独立加密；撤销一台 Device 不要求其他 Device 轮换群组密钥。
- 接收方能验证发送方身份、目标 Device、Installation 和消息完整性。
- LAN 与 Relay 发生故障切换时，Remote Command 可以保留相同 `command_id` 并安全重试。
- Go daemon 与 TypeScript/Expo 客户端产生逐字节兼容的消息。
- 未知版本、过期、重放、错误路由和认证失败具有稳定机器错误码。

## 密码套件

`suite` 固定为：

```text
HPKE-X25519-HKDF-SHA256-CHACHA20POLY1305+Ed25519
```

其组成如下：

- 公钥加密：RFC 9180 HPKE Base mode。
- KEM：`DHKEM(X25519, HKDF-SHA256)`，标识 `0x0020`。
- KDF：`HKDF-SHA256`，标识 `0x0001`。
- AEAD：`ChaCha20Poly1305`，标识 `0x0003`。
- 发送方签名：RFC 8032 Ed25519。
- 二进制文本表示：无 padding 的 RFC 4648 base64url。

HPKE `info` 是下列 UTF-8 字节，末尾包含一个 NUL 字节：

```text
Herdr Connect Protocol v1 HPKE\0
```

每个身份分别拥有一对 X25519 加密密钥和一对 Ed25519 签名密钥。两种密钥不能复用、转换或从同一个 seed 派生。公钥和私钥在协议边界均使用 32 字节原始格式；Ed25519 私钥使用 32 字节 seed，而不是 64 字节展开形式。

## Envelope

外层 JSON 只有三个字段：

```json
{
  "protected": "base64url(受保护头 UTF-8 字节)",
  "ciphertext": "base64url(HPKE ciphertext)",
  "signature": "base64url(Ed25519 signature)"
}
```

外层 JSON 自身不参与签名，不能添加业务字段。所有参与路由、期限和版本判断的数据都必须位于 `protected` 中。

### 受保护头

`protected` 解码后必须是无 BOM、无空白、无重复字段、无未知字段的 UTF-8 JSON。字段必须严格按下列顺序出现，所有字段始终存在；不适用的领域字段使用空字符串或 `0`：

```json
{
  "v": 1,
  "suite": "HPKE-X25519-HKDF-SHA256-CHACHA20POLY1305+Ed25519",
  "message_type": "lifecycle_event",
  "installation_id": "ins_example",
  "sender_id": "installation_primary",
  "sender_signing_key_id": "sign_installation_1",
  "sender_signing_public_key": "",
  "recipient_id": "device_iphone",
  "recipient_encryption_key_id": "enc_device_1",
  "message_id": "msg_example",
  "event_id": "evt_example",
  "event_seq": 42,
  "through_event_seq": 0,
  "command_id": "",
  "request_id": "",
  "ack_seq": 0,
  "created_at_ms": 1784084400000,
  "expires_at_ms": 1784088000000,
  "enc": "base64url(HPKE encapsulated key)"
}
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `v` | 协议主版本，当前只能是 `1` |
| `suite` | 固定密码套件名称 |
| `message_type` | 本文定义的消息类型 |
| `installation_id` | 授权与序列空间所属 Installation |
| `sender_id` | 发送方 Installation 或 Device 身份 |
| `sender_signing_key_id` | 选择已信任 Ed25519 公钥的非秘密标识 |
| `sender_signing_public_key` | 仅 `pairing_request` 使用的候选 Device Ed25519 公钥；其他消息固定为空字符串 |
| `recipient_id` | 唯一目标身份，不允许广播解密 |
| `recipient_encryption_key_id` | 选择目标 X25519 公钥的非秘密标识 |
| `message_id` | 本次 envelope 唯一身份，用于传输重放防护 |
| `event_id` | Lifecycle Event 的逻辑身份；逐 Device 加密时保持相同 |
| `event_seq` | Installation 分配的严格递增事件序号 |
| `through_event_seq` | State Snapshot 已覆盖到的事件序号 |
| `command_id` | Remote Command 在重试和换路时保持不变的身份 |
| `request_id` | Output Request 与对应 Snapshot 的关联身份 |
| `ack_seq` | Device 已完整处理的连续事件 cursor |
| `created_at_ms` | Unix epoch 毫秒整数 |
| `expires_at_ms` | 到期时刻；到达该毫秒即无效 |
| `enc` | RFC 9180 HPKE encapsulated key |

所有标识符长度为 1–128 个 ASCII 字符，匹配：

```text
^[A-Za-z0-9][A-Za-z0-9._:-]{0,127}$
```

所有序号和 Unix 毫秒时间必须是 `0` 到 `2^53-1` 范围内的整数，从而保证 Go 与 JavaScript 无损互操作。语义上要求存在的序号从 `1` 开始。

### 加密与签名顺序

发送方执行：

1. 校验头、消息类型、TTL 和大小限制。
2. 创建 HPKE sender context，取得 `enc`。
3. 按固定字段顺序编码受保护头。
4. 以受保护头原始字节作为 HPKE AAD，加密 plaintext。
5. 构造签名输入并用发送方 Ed25519 私钥签名。
6. 将三个二进制部分编码为无 padding base64url。

签名输入为以下字节拼接：

```text
UTF8("Herdr Connect Protocol v1 signature\0")
|| uint32_be(len(protected_bytes))
|| protected_bytes
|| ciphertext_bytes
```

接收方先严格解析并检查版本、suite、路由和大小，再验证 Ed25519；只有签名通过后才执行 HPKE open。签名或 HPKE 认证失败统一返回 `authentication_failed`。

HPKE Base mode 的 AEAD nonce 由 RFC 9180 key schedule 和单次 sender context 的 sequence number `0` 派生，不在线上单独传输。每个 envelope 必须创建新的 sender context 和 `enc`，不得复用；篡改 `enc` 会改变密钥与 nonce 上下文，并最终收敛为 `authentication_failed`。

## 消息类型与期限

| `message_type` | 最大 TTL | 额外受保护头要求 |
| --- | ---: | --- |
| `session_hello` | 30 秒 | 无 |
| `pairing_request` | 5 分钟 | `sender_signing_public_key` 是 32 字节公钥的 base64url |
| `pairing_decision` | 5 分钟 | 无 |
| `lifecycle_event` | 24 小时 | `event_id` 非空，`event_seq > 0` |
| `state_snapshot` | 5 分钟 | 使用 `through_event_seq` |
| `output_request` | 30 秒 | `request_id` 非空 |
| `output_snapshot` | 30 秒 | `request_id` 非空 |
| `remote_command` | 30 秒 | `command_id` 非空 |
| `command_result` | 5 分钟 | `command_id` 非空 |
| `ack` | 5 分钟 | `ack_seq > 0` |
| `error` | 5 分钟 | 无 |

`expires_at_ms` 必须大于 `created_at_ms`，且差值不能超过表中上限。接收方允许 `created_at_ms` 最多比本机时间超前两分钟，用于识别明显错误时钟；但不对 `expires_at_ms` 增加宽限，到达到期时刻即拒绝。Remote Command 执行层仍须使用本机接收时间和在线状态实施不超过 30 秒、无离线队列的规则。

## Payload

Payload 是 HPKE plaintext，使用 UTF-8 JSON。字段采用 `snake_case`。本节定义 v1 的领域形状；实现可以增加纯本地类型，但不能在线上增加未定义字段。

### `session_hello`

```json
{
  "supported_versions": [1],
  "last_ack_event_seq": 41,
  "session_nonce": "opaque-random-id"
}
```

### `pairing_request`

```json
{
  "pairing_id": "pair_example",
  "pairing_secret": "base64url(32 random bytes)",
  "device_name": "Alice 的 iPhone",
  "device_signing_public_key": "base64url(32 bytes)",
  "device_encryption_public_key": "base64url(32 bytes)",
  "challenge": "base64url(32 random bytes)"
}
```

二维码只包含 `server_url`、`v`、一次性 `pairing_id`、五分钟随机 secret 和 Installation 公钥短指纹。Device 从本地配对端点取得 Installation 公钥并核对指纹。桌面确认前不得建立持久授权。

首次请求不存在预先受信任的 Device 公钥，因此采用下列专用自举顺序：

1. Device 将由自身 Ed25519 私钥导出的公钥写入受保护头 `sender_signing_public_key`，并照常对整个 envelope 签名。
2. daemon 严格解析头、校验路由与期限，使用该候选公钥验签，通过后才执行 HPKE open。该自签名只证明同一请求持有对应私钥，不代表 Device 已获授权。
3. daemon 调用 Go `ValidatePairingCandidate` 或 TypeScript `validatePairingCandidate` 严格解析 plaintext，要求 `device_signing_public_key` 与受保护头中的候选公钥逐字节相等，并校验 `pairing_id`、32 字节 `pairing_secret`、两把 32 字节 Device 公钥、1–128 字符设备名和 32 字节 `challenge`。未知字段、错误类型或错误长度都必须拒绝。
4. 协议 `Open` 只在解密和 validator 成功后调用专用 `PairingGuard`。guard 必须在同一个本地事务中记录 `message_id`、验证并消耗一次性 pairing session，再进入电脑端待确认状态；配对请求不经过普通 `ReplayGuard`。secret 错误、payload 无效时不得调用 guard，session 已使用、失败次数超限或超过五分钟都必须拒绝。

普通消息以及 `pairing_decision` 不允许携带 `sender_signing_public_key`，必须使用调用方按 `sender_signing_key_id` 选择的已信任公钥。协议库的 `Open` 仅允许 `pairing_request` 在未传入预信任发送方公钥时使用上述自举分支。

validator 返回的是 `PairingCandidate`，不是已授权 Device。它不写授权状态、不分配 mailbox，也不允许调用远程能力；`PairingGuard.AcceptIfNew`（TypeScript 为 `acceptIfNew`）是原子提交接缝。daemon 只有在 validator 成功、该 guard 原子消耗 session 且电脑端二次确认后，才能在另一个明确的确认事务中创建已授权 Device。

### `pairing_decision`

```json
{
  "pairing_id": "pair_example",
  "decision": "accepted",
  "installation_signing_public_key": "base64url(32 bytes)",
  "installation_encryption_public_key": "base64url(32 bytes)",
  "challenge_signature": "base64url(64 bytes)"
}
```

`decision` 只能是 `accepted` 或 `rejected`。

`challenge_signature` 使用 Installation Ed25519 私钥签署下列 transcript。`field(x)` 表示 `uint32_be(len(x)) || x`；文本先编码为 UTF-8，base64url 字段先解码为原始字节：

```text
UTF8("Herdr Connect Protocol v1 pairing challenge\0")
|| field(pairing_id)
|| field(pairing_secret_raw_32)
|| field(challenge_raw_32)
|| field(device_signing_public_key_raw_32)
|| field(device_encryption_public_key_raw_32)
|| field(installation_signing_public_key_raw_32)
|| field(installation_encryption_public_key_raw_32)
|| field(decision)
```

Device 必须先用 QR 指纹确认过的 Installation 签名公钥验证该签名，再接受 `decision`。外层 `pairing_decision` envelope 仍须由同一 Installation 公钥验签并加密给候选 Device。`accepted` 只在电脑端二次确认后产生；`accepted`、`rejected`、超时和失败次数超限都会永久消耗该 pairing session，后续相同 `pairing_id` 返回重放或失效错误，不能重新授权。

Go 通过 `PairingChallengeTranscript`、`SignPairingChallenge`、`VerifyPairingChallenge` 实现该 transcript；TypeScript 对应 `pairingChallengeTranscript`、`signPairingChallenge`、`verifyPairingChallenge`。两端都必须先验证字段长度和 `decision`，验签失败统一返回 `authentication_failed`。

### `lifecycle_event`

```json
{
  "agent_id": "agent_1",
  "turn_id": "turn_42",
  "lifecycle_revision": 8,
  "interaction_state": "blocked",
  "turn_outcome": null
}
```

`interaction_state` 为 `working`、`blocked`、`ready_input`、`unknown`。`turn_outcome` 为 `succeeded`、`failed`、`cancelled` 或 `null`；它与交互状态独立。

### `state_snapshot`

```json
{
  "agents": [
    {
      "agent_id": "agent_1",
      "turn_id": "turn_42",
      "lifecycle_revision": 8,
      "interaction_state": "blocked",
      "turn_outcome": null
    }
  ]
}
```

快照覆盖位置来自受保护头 `through_event_seq`。Device 接收快照后丢弃不高于该位置的旧增量，再从后续连续事件继续。

### `output_request` 与 `output_snapshot`

```json
{
  "agent_id": "agent_1",
  "expected_turn_id": "turn_42",
  "expected_lifecycle_revision": 8
}
```

```json
{
  "agent_id": "agent_1",
  "turn_id": "turn_42",
  "lifecycle_revision": 8,
  "text": "最近输出……",
  "line_count": 73,
  "truncated": true
}
```

Output Snapshot 最多 200 行且 plaintext 最多 256 KiB，只加密给请求 Device，不形成 Relay 历史。

### `remote_command`

发送 prompt：

```json
{
  "operation": "send_prompt",
  "agent_id": "agent_1",
  "expected_turn_id": "turn_42",
  "expected_lifecycle_revision": 8,
  "text": "继续，并先检查失败测试。"
}
```

中断：

```json
{
  "operation": "interrupt",
  "agent_id": "agent_1",
  "expected_turn_id": "turn_42",
  "expected_lifecycle_revision": 8
}
```

`send_prompt` 的 `text` 是最多 16 KiB 的 UTF-8 纯文本。MVP 不允许附件、文件、语音、富文本或工作中排队。`interrupt` 必须调用 Herdr 结构化接口，不能转换为按键注入。

### `command_result`

```json
{
  "status": "rejected",
  "error_code": "stale_turn",
  "current_turn_id": "turn_43",
  "current_lifecycle_revision": 1
}
```

`status` 为 `accepted` 或 `rejected`。相同 `command_id` 的重试返回原结果，不重复产生副作用。

### `ack` 与 `error`

`ack` payload 是空对象 `{}`；连续 cursor 位于 `ack_seq`。通用错误 payload：

```json
{
  "error_code": "gap_requires_snapshot",
  "retryable": true
}
```

## 重放与排序

- 只有签名、期限和 HPKE 认证全部通过后，接收方才原子记录 `message_id`。
- `message_id` 至少保留到 `expires_at_ms`。重复值返回 `replay`。
- Remote Command 另外按 `command_id` 做持久幂等；传输换路不能生成新 `command_id`。
- Lifecycle Event 的 `event_seq` 由 daemon 在 Installation 范围内持久、原子、严格递增分配。
- 每个 Device 独立 ACK。发现 gap 时必须请求 State Snapshot，不能猜测缺失状态。
- 同一事件面向不同 Device 生成不同 ciphertext，但 `event_id` 与 `event_seq` 相同；`message_id` 可以按 envelope 分配。

## 大小限制

- plaintext：最多 256 KiB。
- HPKE ciphertext：最多 256 KiB 加 16 字节 AEAD tag。
- 受保护头：最多 4096 字节。
- Ed25519 signature：固定 64 字节。
- HPKE `enc`：固定 32 字节。

更严格的消息级限制仍然有效，例如 prompt 的 16 KiB 和 Output Snapshot 的 200 行。

## 稳定协议错误码

| 错误码 | 含义 |
| --- | --- |
| `unsupported_version` | `v` 不是 1 |
| `unsupported_suite` | 密码套件不匹配 |
| `unsupported_message_type` | 未知消息类型 |
| `invalid_envelope` | base64url、JSON、字段集合或 canonical encoding 无效 |
| `invalid_header` | 标识符、序号或消息专属字段无效 |
| `invalid_key` | 本地提供的原始密钥格式无效 |
| `wrong_route` | Installation、sender 或 recipient 与调用方预期不一致 |
| `authentication_failed` | 签名或 HPKE 认证失败，不进一步区分 |
| `created_in_future` | 创建时间超过允许偏差 |
| `expired` | 已到 `expires_at_ms` |
| `ttl_exceeded` | 声明期限超过消息类型上限 |
| `replay` | `message_id` 已接受过 |
| `replay_store_failed` | 无法原子读取或写入重放状态 |
| `message_too_large` | 受保护头、plaintext 或 ciphertext 超限 |

## 版本演进

- v1 接收方必须拒绝未知主版本，不能猜测或降级解析。
- 增加或删除受保护头字段、改变字段顺序、密码 suite、签名输入、消息类型语义或 payload 必填字段，均需要新主版本。
- 只增加实现内部信息、日志字段或传输层元数据不改变 wire protocol。
- 协商发生在 `session_hello`；不存在共同版本时返回 `unsupported_version` 并终止会话。
