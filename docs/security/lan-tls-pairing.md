# LAN TLS and Pairing

This document describes the security model of the `demo-lan` LAN transport introduced for issue #21. It is the companion to the [Protocol v1 threat model](protocol-v1-threat-model.md): where Protocol v1 specifies message-layer end-to-end encryption for a future relay-based deployment, this document specifies the connection-layer security that ships today for same-LAN use.

It is not a "it uses TLS, therefore it is safe" claim. It states explicitly which attackers are considered, which assets are protected, which risks remain, and where the implementation knowingly deviates from the original issue #21 design.

## Scope

This covers the HTTPS server in `internal/demolan`, the certificate identity and pairing logic in `internal/lanauth`, and the `herdr-connect pair` CLI in `internal/daemoncli`. It does not cover the mobile client (pinned-fetch module and pairing UI), which is documented separately once it lands.

## Trust model

The LAN is untrusted. Other hosts on the same LAN can observe mDNS, scan ports, and block, delay, replay, or modify traffic. mDNS advertises only opaque service metadata: the `_herdr-connect._tcp` service type, the port, a `path` TXT key, a `api_version` TXT key, and an `fp` TXT key carrying the installation certificate fingerprint. Discovery proves reachability only; it does not establish trust or grant access.

Protection is split into two layers:

1. **Connection-layer TLS with certificate fingerprint pinning.** The server presents a self-signed ECDSA P-256 certificate. Devices pin the SHA-256 fingerprint of the certificate's DER encoding and do not perform standard CA-chain or hostname validation. A TLS handshake against an attacker who lacks the private key cannot succeed; a device that has a stored fingerprint detects any certificate change at the TLS layer.
2. **Per-device bearer tokens.** Every endpoint except `/v1/pair` requires `Authorization: Bearer <token>`. Tokens are issued only through pairing, stored only as SHA-256 hashes, and checked for revocation on every request. Unpaired, unknown, or revoked requests receive a structured `401 unauthorized` response.

This is connection-layer security, not message-layer end-to-end encryption. It defends against passive observers and active MITM on the LAN, and against unpaired devices on the same LAN. It does not defend against a fully compromised installation host: the host can read restricted Agent output and execute allowed input, so Protocol v1 cannot protect Agent content once the host falls either.

### Why not message-layer E2EE today

Protocol v1 specifies per-device HPKE with Ed25519 sender signatures and X25519 recipient keys, designed for an untrusted relay (a Cloudflare Worker) that should not be able to decrypt payloads. That machinery is not integrated in the current milestone. It is reserved for the future relay milestone, where the relay can observe and modify ciphertext.

For same-LAN use, connection-layer TLS pinned to a stable installation identity provides equivalent protection against the LAN attacker at materially lower complexity, and it composes cleanly with standard platform TLS stacks (iOS `URLSession` server-trust evaluation). The design intentionally leaves a conceptual upgrade path to Protocol v1 identities rather than building a throwaway mechanism; see [Relationship to Protocol v1 identities](#relationship-to-protocol-v1-identities).

## Certificate lifecycle

The installation identity is a self-signed certificate generated by `lanauth.LoadOrCreateCertificate` and stored as `lan-cert.pem` / `lan-key.pem` next to the database:

- **Algorithm:** ECDSA on the NIST P-256 curve (`elliptic.P256`).
- **Subject:** CN `Herdr Connect LAN`, no SANs. Devices pin the fingerprint and do not validate hostnames, so SANs would add no value.
- **Validity:** 10 years from generation (`certValidityYears = 10`). It is not rotated. Rotation would force every paired device to re-pair, so the certificate is treated as a long-lived installation identity, not a routinely cycling credential.
- **Pinning target:** the SHA-256 of the leaf certificate DER, encoded with base64 RawURL when rendered into the QR payload or returned from `/v1/pair`.
- **Validation policy:** devices compare the observed leaf certificate fingerprint to the stored value in constant time during server-trust evaluation. Standard CA-chain validation is not used, because the certificate is self-signed.
- **Concurrency:** if the `pair` CLI and the `daemon` service both attempt a first-time generation, they race on `O_CREATE|O_EXCL` for the private key file; the loser waits for the winner to finish writing both files. Both processes end with the same identity.
- **Regeneration equals re-pairing.** If the certificate files are deleted or regenerated, the fingerprint changes and every previously paired device must pair again. Devices that stored the old fingerprint report a fingerprint mismatch and cannot connect until re-paired.

The private key is written with mode `0600` (owner-only) and the certificate with `0644`. Permissions are mirrored for the platform in `permissions_unix.go` / `permissions_windows.go`.

## Pairing model

Pairing establishes a per-device token. The flow is:

1. The owner runs `herdr-connect pair` while `demo-lan` is running.
2. `pair` generates a 32-byte one-time secret, stores only its SHA-256 hash with a 5-minute TTL (`PairingSecretTTL`), and renders the plaintext secret into a terminal QR alongside the fingerprint, candidate host addresses, and port.
3. A device scans the QR and submits the plaintext secret plus a device name to `POST /v1/pair`.
4. The server completes pairing in a single SQLite transaction: it conditionally consumes the secret (only if it exists, is unconsumed, and has not expired), inserts a new paired device with a fresh per-device token stored as a SHA-256 hash, and links the device to the secret. The plaintext token is returned exactly once in the pairing response and is never persisted or logged in plaintext.
5. `pair` polls the secret status and exits once the secret is consumed, printing the paired device name and `device_id`.

Failure modes are deliberately uniform to avoid leaking state:

- An invalid pairing request (malformed JSON, empty device name or secret) returns `400 invalid_pairing_request`.
- A secret that is absent, expired, or already consumed returns the same `400 pairing_secret_invalid`. The server does not distinguish these cases, so an unauthenticated caller cannot probe secret status.
- Any other request to a non-pairing endpoint without a valid bearer token returns `401 unauthorized`, without distinguishing a missing token from an unknown one. A revoked device's token is deliberately carved out of this uniformity — it returns `401 revoked` instead — because knowing a token was revoked is only actionable information to whoever already holds that exact 32-byte token (the previously-paired device itself); an attacker without the token gains nothing from this, since guessing a valid token remains computationally infeasible either way. This lets the mobile client show "this device was revoked, pair again" instead of a generic authentication failure. See issue #22.

### Automatic approval is a known deviation

Issue #21 as originally written required interactive confirmation on the host before pairing completed. The shipped implementation approves pairing automatically: a device that submits a valid one-time secret is paired immediately, with no second click on the host.

This deviation is intentional and scoped to the current single-user milestone. The one-time secret is rendered into a QR that is only legible to someone physically in front of the host's screen, so scanning the QR is itself an out-of-band confirmation that the owner intends to pair this device. In a single-user deployment that out-of-band act carries the same weight as a host-side confirmation prompt, without requiring the owner to return to the host keyboard while holding the device. The deviation is recorded here and in issue #21 rather than reversed silently.

When multi-user or unattended deployments become a real requirement, the natural extension is to add an explicit confirm step between secret consumption and token issuance; the single-transaction `CompletePairing` path is the seam where that gate would go.

## Revocation

`lanauth.RevokeDevice` sets `revoked_at_ms` on a paired device. Once revoked, the device's bearer token no longer authenticates: `Authenticate` returns `AuthStatusRevoked`, which the HTTP layer maps to a `401 revoked` response distinct from the generic `401 unauthorized` for missing or unknown tokens. Revocation is host-side only; there is no remote recovery or key escrow in this milestone, matching the Protocol v1 stance that a lost or compromised device must be revoked from the installation host.

A CLI for listing and revoking paired devices is available via `herdr-connect devices list` and `herdr-connect devices revoke <device_id>`.

## Relationship to Protocol v1 identities

Protocol v1 models pairing as establishing an `Installation` and `Device` long-term identity exchange, backed by Ed25519 signing keys and X25519 HPKE recipient keys. The current LAN transport is conceptually the same pairing step — one installation, one or more devices, each with a revocable per-device credential — but substitutes implementation primitives:

| Concept | Protocol v1 | LAN transport (today) |
|---|---|---|
| Installation identity | Ed25519 / X25519 keypair | Self-signed ECDSA P-256 certificate, pinned by SHA-256 fingerprint |
| Device credential | Per-device keypair + signed grant | Per-device bearer token (random 32 bytes, stored as SHA-256) |
| Pairing bootstrap | Out-of-band identity exchange | One-time QR secret, 5-minute TTL, single consumption |
| Transport protection | Per-message HPKE over a relay | Connection-layer TLS pinned to the fingerprint |
| Revocation | Signed revocation + key rotation | Host-side `revoked_at_ms` checked on every request |

The mapping is intentional: both models reduce to "installation has a stable identity, devices hold revocable credentials granted at pairing time." The design goal is that a future migration to Protocol v1 identities reuses the same pairing concept and the same device/revocation storage shape, replacing the certificate-fingerprint identity with an Ed25519/X25519 identity without forcing users to relearn the model.

No forward compatibility is promised. Devices that pair against today's certificate-fingerprint identity may need to re-pair when Protocol v1 identities land; the storage and CLI seams are shaped to allow migration, not to guarantee it without re-pairing. This section records the design intent so the future milestone can evaluate the trade-off explicitly rather than rediscovering it.

## What is still out of scope

- **Multi-user or unattended host approval.** Pairing is automatic; see [Automatic approval is a known deviation](#automatic-approval-is-a-known-deviation).
- **Certificate rotation.** Certificates are long-lived; rotation equals forced re-pairing.
- **Remote revocation or recovery.** Revocation is host-side only.
- **Message-layer E2EE over a relay.** That is the Protocol v1 milestone; the current transport is connection-layer only.
- **TXT-record-based fingerprint discovery.** The `fp` TXT key is published for future clients, but the current mobile discovery library cannot read mDNS TXT records, so fingerprint matching relies entirely on the value captured at pairing time plus TLS-layer pinning on every request. A certificate change is detected as a TLS handshake failure, not via TXT comparison.
